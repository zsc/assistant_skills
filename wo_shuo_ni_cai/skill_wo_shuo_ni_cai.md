# 1. 技能目标

把“我说你猜”做成一个**边界清晰、可复用、可评测、可抢占恢复**的独立 skill，而不是塞进开放域大提示词里。

本 skill 的目标不是“像人一样随便猜”，而是：

1. 把用户心中目标映射到一个**稳定、常见、可二分收敛**的候选空间。
2. 在不依赖屏幕的前提下，用**短句、单问题、可口头回答**的方式持续收窄候选。
3. 在第 8 轮内尽量收敛到**单一候选**，并用 yes/no 方式完成最终确认。
4. 在车端场景下，能被导航/靠边停车等高优先级能力**随时抢占**，事后可继续。

---

# 2. 适用场景

适用：

- 用户主动发起小游戏：如“来玩我说你猜”“你来猜我想的东西”“玩个 yes/no 猜东西”。
- 语音陪伴或轻社交场景，尤其是停车等待、低负荷通勤、闲聊插空。
- 需要证明 AI 在开放域聊天里“能接住话”，但又希望能力边界明确、可测试、可迭代。

不适用：

- 驾驶任务繁忙、连续导航播报、靠边停车执行中、危险告警期间。
- 用户要玩的其实是“猜谜语”“看描述猜答案”“狼人杀式推理”——那是别的 skill。
- 用户选择的是**极冷门、私密、动态变化、无法客观回答 yes/no** 的目标。
- 目标涉及明显不适合在车端游戏化处理的内容（自伤、爆炸物、未成年人色情、违法交易等）。

# 3. 用户契约（进入游戏时必须讲清楚）

默认玩法：

- 你先在心里想好一个目标。
- 最好给我一个**粗提示**，例如“动物 / 食物 / 日用品 / 一个人物 / 跟车有关”。
- 这个粗提示**不计轮次**。
- 我每次只问一个问题。
- 你只回答“是 / 否 / 不确定”。
- 我在 8 轮内猜中算我赢，否则你赢。

用户需要遵守：

1. 目标在游戏开始前就确定，不能中途改。
2. 尽量选择**大众可理解的稳定名词**，而不是私密梗、缩写、昨天梦见的东西、或“我现在的心情”。
3. 回答以“这个目标的最常见、最典型理解”为准。
4. 如果问题确实无法判断，可以回答“不确定”，但不建议频繁使用。

推荐目标范围：

- 动物、植物、食物、饮品、日用品、工具、电子产品、交通工具、地点、职业、公众人物、影视角色、品牌、建筑、自然物。

限制目标范围：

- 私人联系人、身份证号、车牌号、随机字符串、一次性事件、瞬时状态、最新热搜、当前职位持有人、个人隐私、复合答案（比如“手机和钥匙一起”）。

# 4. 胜负定义

- **AI 胜**：在第 1~8 轮中的任意一轮，AI 用单一候选发起确认，用户回答“是”。
- **用户胜**：
  - 第 8 轮后 AI 仍未命中；或
  - 第 8 轮 AI 发起最终确认但用户回答“否”；或
  - 用户因前述规则限制导致游戏无法公平继续，选择重开，则本局不计胜负。

轮次定义：

- 一轮 = AI 提出一个可用“是/否/不确定”回答的问题 + 用户给出一个有效回答。
- AI 的“最终猜测”也必须以 yes/no 问句表达，例如：
  - “你想的是雨伞吗？”
  - “这个目标是特斯拉吗？”
- 因此，**最终猜测也计入轮次**。

# 5. 技能入口与路由

## 5.1 显式触发词

- “玩我说你猜”
- “你来猜我想的东西”
- “玩个猜东西”
- “来个 yes/no 猜谜游戏”
- “你问我答，8轮内猜中”

## 5.2 隐式触发条件

当满足以下条件时可从开放域聊天切入：

- 用户表达出“我心里想了一个东西，你来猜”的明确意图；
- 用户接受 yes/no 规则；
- 当前车端状态允许低优先级互动。

## 5.3 抢占与恢复

若出现以下高优先级事件，立即暂停 skill：

- 安全告警
- 模糊导航新指令
- 语音靠边停车请求
- 来电 / 系统关键通知

暂停时保存：

- 当前轮次
- 已问问题与回答
- 当前候选簇 / top-3 猜测
- 是否已进入 final-guess 模式

恢复时使用一句话接回：

- “我们继续，当前是第 4 轮。它通常能放进口袋吗？”

# 7. 状态机

```text
idle
  -> setup                # 解释规则，邀请用户想好目标
  -> waiting_hint         # 用户决定给不给粗提示
  -> playing              # 正常问答轮
  -> repair               # 回答冲突 / 无效回答修复
  -> final_guess          # 候选已充分收敛，进入单一猜测确认
  -> won                  # AI 猜中
  -> lost                 # 8轮未中
  -> paused               # 被高优先级能力打断
  -> restart              # 用户要求重开
  -> exit                 # 用户退出或场景不适合继续
```

状态转换规则：

- `setup -> waiting_hint`：用户接受规则。
- `waiting_hint -> playing`：获得粗提示，或用户明确表示“不提示也玩”。
- `playing -> repair`：发现明显前后矛盾，或用户回答非 yes/no。
- `playing -> final_guess`：top-1 候选足够高，或剩余有效候选 <= 3，或已到第 7 轮。
- `final_guess -> won`：用户答“是”。
- `final_guess -> lost`：用户答“否”且已是第 8 轮。
- `* -> paused`：高优先级能力抢占。
- `paused -> playing/final_guess`：用户说“继续”。

# 8. 核心策略：先分桶，再打特征，再单点确认

## 8.1 设计原则

8 个 yes/no 问题最多区分 256 个叶子节点，因此不能把世界当成均匀空间暴力二分。这个 skill 必须依赖：

1. **粗提示降低先验空间**；
2. **常识高频候选优先级**；
3. **领域化特征提问模板**；
4. **后期尽早转单点确认**。

换句话说，这不是“通用 20 Questions”，而是“对常见目标做高效收敛”。

## 8.2 总体收敛顺序

优先顺序：

1. 确定大类
2. 确定是否为现实实体 / 虚构对象
3. 确定典型使用场景 / 栖息环境 / 功能
4. 确定尺寸 / 可携带性 / 是否可食用 / 是否有生命等强区分特征
5. 锁定子类
6. 在第 6~8 轮完成单一候选确认

## 8.3 候选评分（概念层）

内部不要求精确数值推理，但需要遵循以下排序思想：

```text
candidate_score
= commonness_prior
* hint_match
* answer_consistency
* iconicity
* category_confidence
```

解释：

- `commonness_prior`：大众更常想到的目标优先，例如“猫”高于“薮猫”。
- `hint_match`：与粗提示一致程度越高，分数越高。
- `answer_consistency`：与历史回答冲突越少越高。
- `iconicity`：具有强识别特征的候选优先。
- `category_confidence`：在当前分支下更像“典型答案”的候选优先。

当满足任一条件时，进入 `final_guess`：

- top-1 明显领先 top-2；
- 估计剩余候选不超过 3 个；
- 已到第 7 轮，必须为第 8 轮的单点确认做准备。

# 9. 提问生成策略

## 9.1 提问硬约束

每个问题必须满足：

1. **单问题**：一次只问一件事。
2. **可口头 yes/no 回答**：用户不需要看屏幕。
3. **客观、稳定**：问常识属性，不问主观感受。
4. **避免双重否定**：例如不要说“它不是不能吃吧？”
5. **避免复合问题**：例如不要说“它是吃的还是喝的？”
6. **避免元问题**：不要问“我是不是快猜到了？”
7. **避免过早报菜名式枚举**：除非已经进入 `final_guess`。
8. **尽量高信息增益**：优先问能把当前候选簇切得最开的特征。

## 9.2 语言风格约束

- 口语化、短句、自然。
- 少用术语，优先大众词。
- 默认中文。
- 问句长度优先控制在 8~18 个汉字。
- 不输出解释性推理。

好的问法示例：

- “它通常是活的吗？”
- “它主要是拿来吃的吗？”
- “它平时能放进口袋吗？”
- “它跟交通出行强相关吗？”
- “它是现实里真实存在的吗？”

不好的问法示例：

- “它是不是动物或者植物或者别的什么？”
- “你是不是想了一个我肯定猜不到但又不是特别冷门的东西？”
- “它不太不是电子产品吧？”

# 10. 分域提问模板（关键）

> 说明：为了提高 8 轮内命中率，skill 不应完全自由发挥，而应优先使用分域模板。

## 10.1 未给粗提示时

建议先争取一个不计轮次的粗提示：

- “先给我一个大类提示，不算轮次，比如动物、食物、日用品、人物。”

若用户拒绝给提示，则首轮优先问：

- “它通常是活的吗？”

原因：对常见目标，活物 / 非活物能快速切分后续模板。

## 10.2 动物 / 生物

优先特征顺序：

1. 家养 / 野生
2. 哺乳类 / 鸟类 / 水生 / 昆虫
3. 体型大小
4. 典型栖息环境
5. 是否常作为宠物 / 食物 / 园区观赏
6. 标志性特征（长鼻子、会飞、条纹等）

问题例子：

- “它通常会被人当宠物养吗？”
- “它主要生活在水里吗？”
- “它一般比猫大很多吗？”
- “它有翅膀吗？”

## 10.3 食物 / 饮品

优先特征顺序：

1. 固体 / 液体
2. 主食 / 零食 / 饮料 / 调味品
3. 甜 / 咸 / 辣
4. 常温 / 热食 / 冷饮
5. 关键原料或典型场景
6. 是否中国语境下高频

问题例子：

- “它通常是喝的吗？”
- “它更常当主食吗？”
- “它一般是甜的吗？”
- “它通常需要加热后吃吗？”

## 10.4 日用品 / 工具 / 电子产品

优先特征顺序：

1. 是否带电 / 电子化
2. 是否手持 / 可穿戴 / 家居固定
3. 是否与清洁 / 通信 / 娱乐 / 工作相关
4. 尺寸与可携带性
5. 是否有屏幕 / 轮子 / 电池 / 线缆等显著特征
6. 家中高频 vs 个人随身

问题例子：

- “它通常需要用电吗？”
- “它平时能放进口袋吗？”
- “它主要是拿来清洁的吗？”
- “它一般有屏幕吗？”

## 10.5 交通工具 / 车相关物

优先特征顺序：

1. 是否用于载人移动
2. 是否有发动机 / 电机
3. 两轮 / 四轮 / 轨道 / 水上 / 空中
4. 个人拥有 vs 公共交通
5. 车内配件 vs 独立交通工具

问题例子：

- “它主要是拿来载人移动的吗？”
- “它通常有四个轮子吗？”
- “它更像车内用品而不是交通工具本体吗？”

## 10.6 地点 / 建筑 / 地理实体

优先特征顺序：

1. 自然 / 人造
2. 国家 / 城市 / 建筑 / 室内场所
3. 中国 / 海外
4. 旅游属性 / 功能属性 / 地标属性
5. 是否大众耳熟能详

问题例子：

- “它是人造建筑吗？”
- “它在中国吗？”
- “它通常是一个城市吗？”
- “它跟旅游观光强相关吗？”

## 10.7 人物 / 公众人物 / 角色

优先特征顺序：

1. 真实人物 / 虚构角色
2. 当代 / 历史
3. 中国 / 海外
4. 娱乐 / 体育 / 政治 / 科技 / 文学
5. 性别
6. 是否以某个代表作品/身份被熟知

问题例子：

- “这是现实中的真实人物吗？”
- “他/她主要因娱乐圈而知名吗？”
- “这是一个虚构角色吗？”
- “这个角色主要来自动画吗？”

# 11. 轮次策略（建议）

并非硬编码，但默认应遵循以下节奏：

## 第 0 步：收提示（不计轮次）

目标：把空间先收缩到一个大类。

默认开场：

- “来吧，你先想好一个目标。最好先给我一个大类提示，这个不算轮次。”

## 第 1~2 轮：切大桶

目标：确定大类 / 现实性 / 核心用途。

要求：

- 不急着报具体名字。
- 每题都要让分支明显减少。

## 第 3~5 轮：切强特征

目标：在已知大类下，利用尺寸、环境、功能、结构等强特征快速收束。

要求：

- 问题要更具体，但仍保持客观。
- 开始形成 top-3 候选。

## 第 6~7 轮：子类锁定

目标：把候选缩到 1~3 个。

要求：

- 若 top-1 已明显高于其他候选，可在第 6 或第 7 轮直接确认。
- 若还不稳，第 7 轮必须问一个能决定第 8 轮最终猜测的问题。

## 第 8 轮：单点确认

硬规则：

- 第 8 轮必须是**单一候选确认**。
- 禁止在第 8 轮再问泛问题。

示例：

- “你想的是雨伞吗？”
- “你想的是猫吗？”
- “这个人物是周杰伦吗？”

# 12. 异常与修复

## 12.1 无效回答

若用户回答不是“是 / 否 / 不确定”，轻量拉回，不要长篇教育：

- “这一轮你只要回答‘是’或‘否’就行。”
- “如果确实难判断，也可以说‘不确定’。”

此类修复默认**不计入轮次**，除非用户已提供了明确有效回答。

## 12.2 前后矛盾

若历史回答冲突明显，可触发一次修复：

- “我这边有点冲突，我重确认一下：它通常需要用电吗？”

修复原则：

1. 一次只修一个关键冲突。
2. 优先修影响最大的特征。
3. 最多连续修复 1 次，避免把游戏玩成审讯。

## 12.3 目标过冷门 / 不可判定

若目标明显不满足玩法公平性：

- “这个目标太偏门了，8 轮很难公平猜中。换一个更常见的试试？”

触发条件示例：

- 私人记忆、圈内梗、极冷门二次元角色、随机 ID、昨天的梦、公司内部代号、最新职位持有人。

## 12.4 用户插科打诨 / 开放域聊天插入

允许接一小句情绪互动，但必须把球拉回本 skill：

- 用户：“你肯定猜不到。”
- AI：“那我认真一点。它通常能放进口袋吗？”

规则：

- 插话承接最多 1 句。
- 不丢失轮次和当前分支。

## 12.5 安全退出

若用户目标可能落入明显不适合的危险或违规内容：

- “这个不适合继续拿来玩猜谜，我们换个普通事物吧。”

# 13. 车端专属要求

## 13.1 语音优先

- 默认按语音第一设计，不依赖屏幕。
- 提问必须短，避免并列从句。
- 不要一次给多个选项让用户记。

## 13.2 低打扰

- 有导航播报时不抢话。
- 若系统判断驾驶负荷高，可拒绝发起：
  - “现在先专注驾驶，等路况轻松些我们再玩。”

## 13.3 可中断可恢复

恢复后无需重置整局，只需：

1. 简述当前轮次；
2. 不复读全部历史；
3. 直接继续下一个高价值问题。

## 13.4 TTS 规范

- 句长短。
- 避免绕口和双重否定。
- 尽量使用高频词，如“活的、吃的、用电、口袋里、车里”。

# 14. few-shot 示例

## 示例 1：日用品（有粗提示）

用户：玩我说你猜。

AI：来吧，你先想好一个目标。最好先给我一个大类提示，这个不算轮次。

用户：日用品。

AI（第 1 轮）：它通常需要用电吗？

用户：否。

AI（第 2 轮）：它平时能放进口袋吗？

用户：是。

AI（第 3 轮）：它主要是拿来清洁的吗？

用户：否。

AI（第 4 轮）：它通常会在下雨天用到吗？

用户：是。

AI（第 5 轮）：你想的是雨伞吗？

用户：是。

AI：我猜中了，这局我赢。

## 示例 2：动物（无粗提示）

用户：你来猜我想的东西。

AI：可以。你先想好一个目标。最好给个大类提示，不算轮次；不提示也能玩。

用户：不提示。

AI（第 1 轮）：它通常是活的吗？

用户：是。

AI（第 2 轮）：它会被很多人当宠物养吗？

用户：是。

AI（第 3 轮）：它一般比猫大很多吗？

用户：否。

AI（第 4 轮）：它经常会叫“汪汪”吗？

用户：否。

AI（第 5 轮）：它经常会叫“喵喵”吗？

用户：是。

AI（第 6 轮）：你想的是猫吗？

用户：是。

AI：被我猜到了。

# 15. 评测指标（上线前必须有）

## 15.1 核心指标

1. **8轮内命中率**
   - overall
   - 按类别拆分：动物 / 食物 / 日用品 / 人物 / 地点 / 交通工具
2. **平均命中轮次**
3. **无效提问率**
   - 复合问题
   - 主观问题
   - 不适合 yes/no 的问题
4. **修复成功率**
   - 矛盾回答后是否能恢复收敛
5. **中断恢复成功率**
6. **用户主观好玩度 / 自然度**

## 15.2 一期建议门槛

在“用户提供粗提示”的前提下：

- overall 命中率 >= 60%
- 动物 / 食物 / 日用品 >= 70%
- 无效提问率 <= 2%
- 中断恢复成功率 >= 95%
- 第 8 轮前进入单点确认的比例 >= 85%

在“无粗提示”的前提下：

- overall 命中率 >= 35%

## 15.3 数据沉淀建议

每局沉淀以下日志字段，作为后续 skill 迭代数据：

```yaml
session_id:
category_hint:
final_target:
won: bool
rounds_used: int
questions:
  - text:
    answer:
    question_type: category | function | environment | size | structure | direct_guess
    valid_binary: bool
    estimated_gain: high | medium | low
interruptions:
repair_count:
user_frustration_signal: low | medium | high
```

这些日志不是为了重放长链路思维，而是为了积累：

- 哪类问题最有效
- 哪些分支最容易走偏
- 哪些目标需要独立子 skill 或额外知识底座

# 16. 与“大提示词”分离的原因

不建议把这个玩法塞进开放域统一大提示词。原因：

1. **交互协议明确**：8 轮、单问题、yes/no、单点确认。
2. **状态显式**：轮次、候选、矛盾修复、暂停恢复都可管理。
3. **可评测**：能做命中率、轮次、无效问题率。
4. **可演进**：后续可替换候选库、分类器、问题策略，而不影响其他聊天能力。
5. **可抢占**：车端高优先级技能能安全中断并恢复。

# 17. 推荐后续拆分的子能力（用于长期模块化积累）

虽然本文件可以单独上线，但长期建议再拆成以下可复用子能力：

1. `coarse_hint_normalizer`
   - 把“跟车有关系”“吃的吧”“像个工具”归一到稳定大类。

2. `binary_question_generator`
   - 输入候选簇，输出高信息增益的 yes/no 问题。

3. `candidate_ranker`
   - 根据粗提示和历史回答维护 top-k 候选。

4. `contradiction_repair`
   - 检测并修复前后回答冲突。

5. `voice_game_recovery`
   - 处理中断后恢复，保持短上下文重启。

这样未来“成语接龙”“你画我猜”“猜人物”“猜地点”都能复用其中一部分，而不是每个玩法重新堆 prompt。

# 18. 建议的首版系统话术

当 skill 被触发时，建议固定首句：

“来吧，你先在心里想好一个目标。最好先给我一个大类提示，比如动物、食物、日用品、人物，这个提示不算轮次。我每次只问一个问题，你只回答‘是’或‘否’，8轮内我猜中算我赢。”

这句的作用：

- 一次性讲清规则；
- 主动争取 coarse hint；
- 把回答格式收紧；
- 不需要用户再追问怎么玩。

# 19. 一句话总结

这个 skill 的本质，不是“会猜”，而是把一个开放域小游戏压缩成：

**明确协议 + 分域提问模板 + 候选收敛状态机 + 车端可抢占恢复**。

这才是适合 grok + fsd 一期里做模块化能力积累的写法。
